# 研究员题目设计逻辑梳理 - Case_001 及通用规则

> 来源：`研究员设计题目思路原稿.txt`
>
> 用途：为后续“题目设计 Agent”开发提供研究员思考逻辑、模块生成规则、题目设计规则、避雷点与 prompt 化素材。

---

## 1. 总体结论

研究员设计 Digital Diary 题目时，并不是从“题目”开始，而是按以下路径推导：

```text
客户商业问题
-> 研究问题拆解
-> 模块结构设计
-> 每个模块内的关键观察点
-> 题目形式与措辞
-> 根据人群、品类、客户已知信息做定制
```

其中，**模块结构决定 60-70 分**，题目措辞和细节决定从 70 分到 80/90 分的提升。

因此，后续 agent 的核心能力不应只是“生成题目”，而应先能：

- 识别客户商业问题。
- 从 Proposal 中读取已经拆好的研究问题。
- 判断哪些模块必须存在，哪些模块可以合并、删减或定制。
- 在每个模块下，用合适题目拿到研究所需信息。

## 2. Case_001 的项目本质

### 2.1 项目类型

Case_001 可以归纳为：

```text
流失/衰退品类寻求增长机会
```

更通用地说，是：

```text
品类增长机会研究
```

### 2.2 商业问题

客户的商业问题来自 brief，不需要研究员擅自改写。

Case_001 中，核心商业问题是：

```text
口香糖品类如何重新增长？如何破局？
```

### 2.3 研究目标

研究目标来自 Proposal 中对商业问题的拆解。研究员在做题目设计时，主要对照 Proposal 中已经拆出的研究问题。

Case_001 的研究问题大致包括：

- 口香糖在真实生活中对应哪些场景与需求？
- 消费者为什么选择或不选择口香糖？
- 益达的 Recenter / 好状态 应如何被理解？
- 绿箭的 After Meal / Fresh & Clean 在真实餐后场景中遇到哪些卡点？
- 口香糖的使用习惯如何形成、中断或变化？
- 品牌和产品层面还有哪些可切入机会？

## 3. 输入材料使用规则

### 3.1 Brief 怎么看

Brief 的价值取决于客户写得有多细。

如果 Brief 只写大目标：

- 不应直接从 brief 写题。
- 应主要看 Proposal，因为 Proposal 已经把商业问题拆成研究问题。

如果 Brief 中已经明确写了客户想问的问题：

- 必须回到 brief 对照。
- 这些问题要在研究设计中体现。
- 可以改写问法或细化，但不能遗漏。

示例：

```text
客户明确问：“今年过年和以往有什么变化？”
-> 研究设计中必须有对应题目。
```

### 3.2 Proposal 怎么看

Proposal 是题目设计最主要的依据。

研究员的逻辑是：

```text
Brief 里的商业问题
-> Proposal 里拆成 1/2/3/4 个研究问题
-> Digital Diary 里每个研究问题变成一个或多个模块
```

因此 agent 在生成题目前，应优先从 Proposal 中提取：

- 项目目标。
- 研究路径。
- 核心研究问题。
- 关键假设。
- 重点场景。
- 重点人群。
- 客户已知信息与待验证问题。

### 3.3 客户既有研究怎么用

客户已经知道的东西，不要重复广撒网。

应当在客户已知基础上继续往下探索。

Case_001 示例：

- 已知益达战略方向是 Recenter / 好状态。
  - 题目就不应泛泛问“什么时候吃口香糖”。
  - 应更聚焦问“状态不佳 / 不自在 / off-center 的时刻”。

- 已知绿箭战略方向是 Fresh & Clean / After Meal。
  - 题目就应聚焦餐后、嘴巴不舒服、口气清新等时刻。

Agent 规则：

```text
如果客户已有明确战略、假设、场景或关键词：
不要重新从零探索；
应围绕这些已知方向进一步拆细真实生活情境、行为、阻碍和机会点。
```

## 4. 模块结构通用规律

研究员认为，Digital Diary 通常遵循一套跨项目的模块骨架。

但不是所有项目都必须完整保留，模块数量与深度取决于：

- 项目目标。
- 品类特性。
- 是否已做过前序研究。
- 客户是否需要深入理解目标人群。
- 受访者负担。
- 礼金与执行成本。

### 4.1 常见模块顺序

```text
模块 1：关于我 / 自我与生活状态
模块 2：典型的一天 / 日常 routine
模块 3：品类/消费图谱 download
模块 4：实时记录 / 日记记录
模块 5：习惯养成 / journey / 变化
模块 6：品牌、产品、未来期待 / 创想
```

### 4.2 模块结构的重要性

研究员原话核心意思：

```text
先把板块画对。
题目多一个少一个，是 70 到 80/90 分的区别；
但板块结构决定至少 60-70 分。
```

Agent 规则：

```text
生成题目前，必须先输出模块结构，并说明每个模块对应哪个研究问题。
```

## 5. 模块 1：关于我 / 自我与生活状态

### 5.1 模块目的

了解受访者是谁，以及 TA 的生活底色。

主要收集：

- 自我描述。
- 职业/专业/人生阶段。
- 居住城市与居住空间。
- 兴趣爱好。
- 生活场景丰富度。
- 社交圈子。
- 生活痛点。
- 理想生活与现实差距。

### 5.2 为什么要问

这些信息用于建立消费者画像，并为后续理解消费场景和需求提供背景。

例如：

- 兴趣爱好越多，生活场景通常越丰富。
- 居住空间能帮助理解生活方式和消费环境。
- 社交圈子能帮助判断 TA 是引领者、跟随者还是普通参与者。
- 理想生活与现实差距可以引出痛点与张力。

### 5.3 常见题目方向

- 请用几个关键词、照片或方式介绍自己。
- 你的工作/学习/生活状态是怎样的？
- 你住在哪里？居住空间是什么样？
- 你有哪些兴趣爱好？哪些长期坚持，哪些最近开始？
- 你的社交生活是怎样的？
- 你理想中的生活是什么样？
- 现实生活和理想生活之间差在哪里？
- 如果给现在生活打分，满分 10 分，你会打几分？扣在哪里？

### 5.4 自评与他评

了解一个人时，可以问：

- 他眼中的自己。
- 别人眼中的他。

但“他评”不是必问。

可以不问的情况：

- 题量已经太多。
- 目标人群很熟悉，如大学生、年轻人。
- 项目不需要做人群分层或社交影响关系分析。

需要问的情况：

- 客户希望深入理解目标客群。
- 品牌希望识别客群中的引领者和跟随者。
- 项目关注社交圈层影响。

### 5.5 痛点怎么问

通用方法：

```text
理想生活是什么？
-> 现实生活和理想生活的差距是什么？
-> 哪些是短期可解决痛点？
-> 哪些是长期社会文化张力？
```

短期痛点可能对应品牌机会。

长期痛点更适合用于理解社会文化趋势。

### 5.6 避雷点

- 不要问过于敏感的隐私问题，如收入、家庭资产等，尤其是高端消费群体。
- 语气要匹配人群：
  - 年轻人可轻松、口语化。
  - 年长或 senior 人群应使用“您”。
- 除最后品牌相关板块外，前面不要暴露客户品牌是谁。

## 6. 模块 2：典型的一天 / 日常 routine

### 6.1 模块目的

收集受访者日常生活节奏和生活场景，用于理解不同消费者类型。

主要收集：

- 起床、工作/学习、通勤、用餐、休闲、睡眠等日常 routine。
- 工作日与休息日差异。
- 情绪变化曲线。
- 与项目相关的重点场景。

### 6.2 Case_001 定制逻辑

Case_001 中，典型一天被两个品牌方向定制：

- 益达关注情绪与状态。
  - 因此增加情绪曲线、状态不佳、不自在等问题。

- 绿箭关注餐后与口气清新。
  - 因此增加用餐、餐前餐后行为、嘴巴清新需求等问题。

### 6.3 通用题目方向

- 请描画一下你典型的一天：几点起床、几点休息、从早到晚都在做什么？
- 工作日和周末有什么不同？
- 一天里哪些时刻最忙、最累、最放松、最容易状态不佳？
- 与本项目相关的场景通常发生在一天中的什么时候？

### 6.4 设计要点

不要把它写成流水账。

坏问法：

```text
早上干什么？中午干什么？晚上干什么？
```

好问法：

```text
请描画一下你典型的一天。我们尤其想了解你在不同时间段的状态、正在做的事情，以及哪些时刻最容易出现和本项目相关的需求。
```

Agent 规则：

```text
典型一天模块必须服务于研究目标。
不能只收集时间流水账；
要收集 routine、状态变化、关键场景、场景背后的需求。
```

### 6.5 用词与顺序

如果客户文件中有高频词或重点词，应优先放在题目提示中。

Case_001 示例：

- 紧张。
- 分心。
- 卡壳。
- 小烦。
- 小累。
- 小乏。

这些词来自客户关注，需要体现在题目提示中，并可根据客户优先级调整顺序。

## 7. 模块 3：品类/消费图谱 download

### 7.1 模块目的

像“打捞”一样，把消费者在相关品类或相邻品类中的消费图谱尽可能收全。

核心是：

```text
先收全图谱，再切分角色、场景、动机和站位。
```

### 7.2 食饮类常见逻辑

如果是食饮项目，需要收集：

- 平时吃什么、喝什么。
- 日常囤的。
- 随手买的。
- 被分享/推荐的。
- 不同场景下吃/喝什么。
- 哪些是常新尝试。
- 哪些是固定消费。
- 不同食饮在生活中扮演什么角色。

### 7.3 其他品类的映射

同一逻辑可迁移到其他品类：

- 笔类：都用哪些笔，最常用、最喜欢、最特别的分别是什么。
- 服饰：衣橱图谱，不同风格、不同场合、不同角色。
- 健康/VMS：保健品、健康设备、健康服务、健康食物、健康习惯。

Agent 规则：

```text
图谱模块不要只问客户品类本身；
应根据研究目标扩展到消费者真实生活中用于解决同类需求的完整解决方案图谱。
```

### 7.4 常用问法

通用方式：

- “最”：
  - 最常用。
  - 最喜欢。
  - 最近新增。
  - 最想复购。
  - 最后悔。

- 场景限定：
  - 工作/学习时。
  - 在路上时。
  - 餐后时。
  - 临上场前。
  - 高消耗/高重复时。

- 客户 guide：
  - 如果客户已有分类或重点场景，跟着客户 guide 问。

- 预设角度：
  - 如果前期已预设某些痛点或属性，应围绕它们追问。

### 7.5 中性词原则

尽量使用中性词，避免过度积极或负面的引导。

Case_001 示例：

推荐：

```text
当你需要调整状态时，你通常会采取哪些方式？
```

避免：

```text
当你有负面情绪时，你会采取哪些应对方式？
```

原因：

- “调整状态”更 general。
- “负面情绪”会引导受访者往消极方向回答。

### 7.6 Guide 的价值

题目前加分类 guide，可以让客户和受访者都有迹可循。

例如：

- 日常囤的。
- 随手买的。
- 被分享/推荐的。
- 餐后时刻。

Agent 规则：

```text
当一个模块要收集多个场景/类型时，应先给出清晰分类 guide，再逐类提问。
```

## 8. 题型选择规则

### 8.1 默认以开放题为主

Digital Diary 前期探索题目通常以开放题为主。

原因：

- 需要获得真实语言。
- 需要发现未知场景、需求和动机。
- 不应过早限制答案空间。

### 8.2 什么时候用选择题

当客户已经给定明确分类，且需要筛选或分流时，可以用选择题。

例如：

- VMS 客户已明确关注 5-7 类健康需求领域。
- 需要先筛出受访者涉及哪些品类或需求，再追问细节。

### 8.3 什么时候用排序题

排序题多用于后段，尤其是：

- 产品概念测试。
- 刺激物测试。
- 多个产品/方向/属性之间的优先级判断。

前期生活方式和场景探索不应过度使用排序题。

### 8.4 素材题使用

照片、视频、语音、截图、实物记录适合用于：

- 需要还原真实场景。
- 需要降低文字负担。
- 需要观察实物、空间、使用方式。
- 目标人群文字表达压力较大。

## 9. 模块 4：实时记录 / 日记记录

### 9.1 模块目的

日记记录主要是为了记录真实场景。

常见记录对象：

- 使用场景。
- 需求触发。
- 进入场景前的准备。
- 当时人物、地点、行动。
- 选择逻辑。
- 使用体验。
- 使用后评价。

### 9.2 记录天数规则

通常至少覆盖：

```text
1 个工作日 + 1 个休息日
```

原因：

- 工作日和休息日生活场景不同。

如果场景和需求非常重要，需要尽可能画全：

```text
可以持续一周记录
```

但记录天数也受：

- 礼金。
- 项目成本。
- 受访者负担。
- 研究目标重要性。

### 9.3 场景记录的标准链路

研究员给出的标准题目链路：

```text
1. 什么时候发生？
2. 当时具体发生了什么？
3. 你在哪里，和谁一起，在做什么？
4. 你怎么形容当时的感受/状态？
5. 你做了什么来应对？
6. 为什么这么做？
7. 有没有选择吃/喝/使用某种产品？
8. 为什么选择或不选择？
9. 如果用了目标品类/品牌，哪个元素真正起作用？
10. 使用前、使用中、使用后感受如何变化？
11. 最后有没有达到你想要的状态？
12. 你怎么评价这个结果？
```

### 9.4 关键原则：不要直接跳到目标品类

坏逻辑：

```text
你状态不好时为什么不吃口香糖？
```

好逻辑：

```text
状态不好时发生了什么？
-> 你做了什么？
-> 有没有吃/喝？
-> 有没有想到口香糖？
-> 为什么用或不用？
```

原因：

口香糖只是众多 coping strategy 的一部分。直接问口香糖会制造预设。

### 9.5 体验追问要拆到元素层

只知道“用了/没用”不够。

如果用了，需要追问：

- 哪个元素起作用？
- 口味？
- 咀嚼过程？
- 清新效果？
- 提神？
- 放松？
- 使用前后状态变化？

可迁移到其他品类：

- 咖啡：咖啡因、口味、香气、油脂、仪式感。
- 护肤：质地、吸收、即时肤感、第二天状态。
- 服饰：面料、颜色、版型、场合适配、他人反馈。

### 9.6 每日题量控制

Case_001 的日记记录已经接近上限。

经验判断：

```text
每天 10-12 个问题左右已是较高负担。
```

Agent 规则：

```text
日记记录必须细，但不能一次丢出过多问题。
每个 bullet 都要有明确目的，避免大而空。
```

## 10. 模块 5：习惯养成 / journey / 变化

### 10.1 模块目的

捕捉变化。

研究员原话核心：

```text
Journey 其实就是问变化。
```

常见关注：

- 原来怎样，现在怎样。
- 哪些吃/用得更多。
- 哪些吃/用得更少。
- 习惯如何形成。
- 如何中断。
- 如何拓宽。
- 如何深入。
- 品类或产品在生活中的角色如何变化。

### 10.2 标准拆法

不要只问：

```text
这个习惯是怎么养成的？
```

应拆成：

```text
第一次尝试是什么时候？
当时为什么尝试？
最初的认知是什么？
后来怎么越来越多/越来越少？
中间发生过什么变化？
有没有中断？为什么？
现在它在你生活里是什么角色？
未来会继续吗？为什么？
```

### 10.3 哪些品类适合问习惯养成

适合：

- 食饮中的成瘾/高频/长期使用品类，如咖啡、酒、电子烟。
- 有品类拓宽路径的品类，如咖啡从速溶到挂耳到现制到咖啡豆。
- 运动、健康习惯、医美等存在明显进入和加深过程的品类。
- 客户关注长期使用、复购、沉迷、流失或忠诚。

不一定适合：

- 普通服饰整体图谱。
- 只做常规产品迭代的品类。
- 习惯养成对商业问题帮助不大的项目。

### 10.4 横向拓宽与纵向深入

Journey 可以分两种：

```text
横向拓宽：从一个品类/形式扩展到更多品类/形式。
纵向深入：在某个细分方向上研究越来越深、投入越来越多。
```

示例：

- 咖啡横向拓宽：速溶 -> 挂耳 -> 现制 -> 咖啡豆。
- 咖啡纵向深入：研究咖啡豆、购买咖啡机、控制萃取方式。

## 11. 模块 6：品牌、产品与未来期待

### 11.1 模块目的

根据客户关注方向，进入品牌、产品或创新机会。

常见方向：

- 品牌印象。
- 品牌关系。
- 品牌变化。
- 品牌人格化。
- 产品期待。
- 产品创新创想。
- 客户预设方向反馈。

### 11.2 品牌怎么问

品牌是抽象概念，因此需要用具体化和 metaphor。

常见问法：

- 你觉得这个品牌像一个什么样的人？
- TA 大概什么年龄？
- TA 的穿搭风格是什么？
- TA 和你的关系亲疏远近如何？
- TA 像你生活里的谁？
- 你怎么形容你和 TA 的关系？
- 你对它的印象有没有变化？什么时候变化的？

适用条件：

```text
只要客户关心品牌问题，就适合这样问。
```

### 11.3 产品怎么问

如果客户更关心产品创新：

- 你还缺一款什么样的产品？
- 如果你是产品设计师，你会设计什么？
- 理想产品在口味、形态、包装、规格、使用方式上是什么样？
- 如果客户已有预设方向，则围绕客户维度追问。

### 11.4 关键原则

最后一板块通常定制化很高。

Agent 规则：

```text
品牌/产品/未来期待模块必须贴客户需求。
如果客户关心品牌，就用 metaphor 具体化品牌。
如果客户关心产品，就用创想题和客户预设维度引导产品机会。
```

## 12. 购买任务 / 使用任务 / 烹饪任务

### 12.1 为什么设置任务

任务型题目用于捕捉真实行为过程中的阻碍、卡点和机会。

常见任务：

- 购买任务。
- 使用任务。
- 烹饪任务。
- 试吃/试用任务。

### 12.2 Case_001 为什么有购物任务

Case_001 中，客户想知道：

```text
口香糖原来人人都吃，现在为什么不吃？
阻碍是否发生在购买过程中？
```

因此设置购物任务，用于观察：

- 什么时候想到买。
- 在哪里买。
- 为什么选择这个渠道。
- 是否比较不同款式。
- 有没有犹豫、麻烦、放弃。
- 买回来放在哪里。
- 是否立刻吃。

### 12.3 什么时候适合设置使用/烹饪任务

适合条件：

- 客户明确关注购买或使用阻碍。
- 产品使用过程本身有复杂步骤。
- 产品创新高度依赖实际使用/烹饪/操作体验。
- 该品类是客户未来重点战略品类。

不一定适合：

- 常规迭代创新。
- 使用过程简单，且不影响核心研究问题。
- 任务会显著增加负担但收益不明确。

### 12.4 是否让 agent 自动决定

研究员认为，是否设置任务不一定适合完全写死在 prompt 中。

更好的方式：

```text
Agent 可以识别并建议是否增加任务模块；
但最终由研究员确认。
```

Prompt 中可写：

```text
如果研究目标涉及购买卡点、使用阻碍、产品创新、烹饪/操作过程，请评估是否需要增加任务型模块，并说明理由与受访者负担。
```

## 13. 语言与措辞规则

### 13.1 贴近目标人群

用词要匹配受访者：

- 年轻人：轻松、口语化。
- 年长人群：礼貌、清楚、少黑话。
- 专业/圈层人群：使用准确圈层语言。

### 13.2 圈层黑话要准确

如果项目涉及圈层或专业品类，应使用受访者圈内准确用语。

例如：

- 攀岩中的路线等级、室内/野外术语。
- 运动、服饰、美妆、医美中的专业说法。

原因：

```text
如果问题一看就是外行，会降低受访者认真回答的意愿。
```

### 13.3 避免品牌暴露

除非到了品牌/产品专项模块，否则前面不应直接暴露客户品牌。

### 13.4 避免预设与诱导

用中性词。

避免：

- 过度负面。
- 过度积极。
- 直接引导到目标品类。
- 暗示客户想听的答案。

## 14. 品类差异化规则

### 14.1 食饮类

重点：

- 场景。
- 需求。
- 情绪。
- 吃喝前中后。
- 竞品/替代方案。
- 触发点。
- 体验元素。
- 习惯养成。

### 14.2 服饰类

重点：

- 衣橱/风格图谱。
- OOTD。
- 场合。
- 搭配过程。
- 犹豫与反复。
- 面料、颜色、版型。
- 希望呈现的感觉。
- 包袋/随身物品。

不一定需要习惯养成模块，除非项目关注某个特定风格或趋势。

### 14.3 健康/VMS 类

重点：

- 健康问题。
- 保健品图谱。
- 健康设备。
- 健康服务。
- 健康食物。
- 健康习惯。
- 信任机制。
- first entry。
- 效果判断。
- 安全性与背书。

### 14.4 美妆/护肤类

取决于项目类型：

- 产品创新：关注包装形式、使用场景、体验、需求缺口。
- 品牌定位：关注品牌关系、印象、人格化。
- 医美相关：关注医美流程、术后修复、护肤品使用变化、趋势。

## 15. Agent 可执行工作流

后续题目设计 Agent 建议按以下步骤工作。

### Step 1：解析输入

从 Brief / Proposal / 客户资料中提取：

- 商业问题。
- 研究目标。
- 目标人群。
- 品类/品牌。
- 客户已知研究沉淀。
- 客户明确关注的场景/需求/关键词。
- 是否有产品创新、品牌、购买阻碍、使用阻碍等重点。

### Step 2：商业问题转研究问题

如果 Proposal 已经拆好，则优先沿用 Proposal。

如果没有拆好，则按以下方式拆：

```text
商业问题是什么？
要回答这个商业问题，需要理解消费者的哪些行为、场景、需求、动机、阻碍、评价和期待？
```

### Step 3：生成模块结构

先输出模块，而不是直接写题。

每个模块必须说明：

- 模块名称。
- 模块目的。
- 对应研究问题。
- 是否通用模块或定制模块。
- 为什么放在这个顺序。

### Step 4：为每个模块生成观察点

先列观察点，再写题。

例如：

```text
模块：日记记录
观察点：
- trigger
- context
- action
- alternative solutions
- choice logic
- target category role
- experience element
- before/during/after change
- evaluation
```

### Step 5：生成题目

题目要：

- 具体。
- 中性。
- 口语自然。
- 贴合人群。
- 贴合品类。
- 避免品牌提前暴露。
- 能引导真实生活细节。
- 必要时要求照片、视频、语音或截图。

### Step 6：质量自检

自检问题：

- 是否回到商业问题？
- 是否覆盖 Proposal 中所有核心研究问题？
- 每个模块是否有明确目的？
- 有没有问客户已经知道且不需要重复的问题？
- 有没有遗漏客户明确要求的问题？
- 是否过早引导到品牌/品类？
- 是否有大而空或流水账问题？
- 是否使用中性词？
- 是否贴合目标人群语气？
- 每日记录题量是否过重？
- 是否有可分析的输出？

## 16. 可直接写进 Prompt 的规则草案

### 16.1 总体规则

```text
你是资深消费者研究员，任务是根据 Brief、Proposal 和客户资料，设计 Digital Diary 问卷题目。

你不能直接从题目开始生成。你必须先：
1. 识别客户商业问题。
2. 提取或拆解核心研究问题。
3. 设计模块结构。
4. 说明每个模块对应的研究问题。
5. 再为每个模块生成题目。
```

### 16.2 输入使用规则

```text
Brief 中客户明确要求的问题必须在设计中体现。
Proposal 中已经拆解的研究问题是模块设计的主要依据。
客户已有研究沉淀不能重复泛问，应在其基础上进一步探索。
```

### 16.3 模块规则

```text
通用模块通常包括：
- 关于我 / 生活状态
- 典型的一天
- 品类/消费图谱
- 实时记录 / 日记记录
- 习惯养成 / journey
- 品牌、产品与未来期待

但你必须根据项目目标、品类、人群、已有研究和受访者负担决定是否合并、删减或定制。
```

### 16.4 题目规则

```text
题目必须服务于研究目的，不得只是流水账。
默认使用开放题，只有在客户已有明确分类或需要筛选/排序时才使用选择题或排序题。
使用中性词，避免诱导。
不要在前期模块暴露客户品牌。
尽量让受访者描述真实场景、具体行为、人物地点、前中后变化和评价标准。
```

### 16.5 日记记录规则

```text
日记记录题必须围绕真实场景展开：
trigger -> context -> action -> alternatives -> choice logic -> target category role -> experience element -> before/during/after -> evaluation

不要直接问“为什么不用目标品类”；
应先问受访者当时做了什么，再自然追问目标品类是否进入解决方案。
```

### 16.6 语言规则

```text
题目语言要匹配目标人群。
年轻人可以轻松口语化；年长或 senior 人群应更礼貌清楚。
如果涉及圈层或专业品类，必须使用准确圈层语言，避免外行表达。
```

### 16.7 模块合并规则

```text
是否合并或压缩“关于我 / 生活状态”和“典型的一天”等基础画像模块，不能只看目标用户画像是否已清楚，必须回到 Proposal 中的研究问题判断。

如果 Proposal 仍然需要通过消费者画像、生活方式、日常场景来解释后续品类场景、需求、机会或品牌切入方向，即使目标用户画像已有一定基础，也仍应保留并适当展开基础画像模块。

如果项目是持续追踪，且前期已经访谈过或已经充分了解目标用户画像，同时本轮 Proposal 不再需要通过画像解释核心研究问题，则可以合并或压缩基础画像模块。

如果客户对目标用户画像已经非常清楚，且本次项目更关注具体场景、节日、campaign、产品或传播问题，则不要重复做完整画像，应将题量集中到客户真正关心的行为、场景和变化上。

Case_001 提醒：虽然客户已有部分人群和品牌研究基础，但 Proposal 仍需要理解消费者生活方式、情绪状态、日常场景与口香糖机会之间的关系，因此“关于我”和“典型一天”仍然分开保留。

示例：百事可乐 CNY 下一步做 campaign，如果客户更关心消费者“怎么过年”，且不关心基础画像，则基础画像模块应压缩。
```

### 16.8 题量与记录天数规则

```text
一般 Digital Diary 总题量可以按 70 题左右作为参考上限。

高端客群或敏感客群需要少而精，例如香奈儿消费者，不要形成骚扰感，整体题量可控制在 50 题左右。

记录天数取决于项目需求、场景覆盖要求和研究员判断。Agent 可以提出建议，但需要提醒由研究员最终确认。
```

### 16.9 Diary 与 IDI 分工规则

```text
Digital Diary 更适合收集浅层 fact 和接近即时的 why：
- 真实发生了什么。
- 当时在哪里、和谁、在做什么。
- 使用/购买/记录了什么。
- 为什么当时这样做。
- 对具体图片、产品图、场景图的直接反应。

IDI 更适合深层、抽象、需要追问的 perception：
- 复杂品牌认知。
- 抽象态度。
- 深层动机。
- 需要研究员持续引导和追问的问题。
- 多个或较长刺激物的理解与比较。
```

### 16.10 刺激物放 Diary 还是 IDI 的规则

```text
如果刺激物是图片、产品图、场景图，信息直接、理解成本低，可以放在 diary 中收集初步反馈。

如果视频数量少、时长短，例如 1-2 个、每个约 1-2 分钟，可以考虑放在 diary。

如果视频较长或数量较多，例如 6 个视频，不适合放 diary，受访者负担过重。

如果刺激物内容复杂，需要引导、追问、澄清认知或深入理解，应放在 IDI。

如果项目本身没有 IDI 设计，Agent 可以提示刺激物测试在 diary 中的风险与负担，并建议研究员确认。
```

## 17. 后续开发建议

### 17.1 Prompt.py 可以拆成 4 层

建议后续 `prompt.py` 不是一整段巨大 prompt，而是模块化：

```text
SYSTEM_ROLE_PROMPT
INPUT_EXTRACTION_PROMPT
MODULE_PLANNING_PROMPT
QUESTION_GENERATION_PROMPT
QUALITY_CHECK_PROMPT
```

### 17.2 Demo 最小闭环

建议 demo 先做成：

```text
上传 Brief + Proposal + 客户资料
-> 自动提取项目信息
-> 输出研究问题拆解
-> 输出模块结构
-> 输出每个模块题目
-> 输出自检报告
```

第一版先不要做 PPT 排版，先做结构化 Markdown 或 JSON 输出。

### 17.3 数据结构建议

每个生成结果至少包含：

```yaml
project_understanding:

  research_objectives:
  target_audience:
  known_client_knowledge:

research_questions:
  - question:
    business_relevance:

modules:
  - name:
    purpose:
    linked_research_questions:
    module_type: generic/custom
    reason_for_order:
    questions:
      - question_text:
        question_type:
        purpose:
        source_logic:
        probe:
        media_requirement:
        caution:

self_check:
  coverage:
  risks:
  missing_info:
  respondent_burden:
```

说明：

```text
这个结构用于 agent 内部思考和 demo 输出，不代表最终平台导入字段必须固定。
不同项目、不同平台、不同执行方式可能需要不同字段。
Agent 可以先判断并建议题型、跳题、素材要求、研究目的、追问说明等信息，但最终是否保留、如何导出，应交由研究员确认。
```

## 18. 补充访谈沉淀规则

以下内容来自后续补充访谈，可直接进入 agent 的模块规划、题量控制和质量自检逻辑。

### 18.1 如何判断模块是否要合并

模块是否合并，核心看“客户是否还需要这部分信息”。

可以合并或压缩的情况：

- 持续追踪项目中，前期已经访谈过受访者。
- 前几轮已经了解过目标用户画像。
- 客户已经很清楚目标用户是谁。
- 本次研究重点不是“理解人”，而是某个具体场景、节日、campaign、产品或传播问题。

示例：

```text
百事可乐 CNY 下一步做 campaign：
如果客户更关心消费者怎么过年，而不关心基础用户画像，
则“关于我”和“典型一天”可以压缩，不需要完整展开。
```

Agent 判断规则：

```text
不要仅凭“目标用户画像已清楚”就压缩基础画像模块。

必须先检查 Proposal：
- 如果核心研究问题仍依赖生活方式、日常场景、情绪状态、价值向往来解释品类机会，则保留基础画像模块。
- 如果核心研究问题已经聚焦到具体节日、campaign、购买任务、产品反馈或后续追踪变化，且画像信息不会显著帮助回答问题，则压缩基础画像模块。
- 如果不确定，应输出“建议保留/压缩”的理由，并标记需要研究员确认。
```

### 18.2 题量和记录天数

通用题量参考：

```text
一般项目：约 70 题左右。
高端/敏感客群：约 50 题左右，少而精。
```

高端客群示例：

```text
香奈儿消费者这类高端客群，不能让受访者感觉被骚扰。
题目需要少而精，减少重复和琐碎追问。
```

记录天数：

- 根据项目需求决定。
- 取决于场景是否需要跨工作日/休息日覆盖。
- 取决于客户是否需要更完整的行为变化。
- 需要研究员最终判断。

Agent 判断规则：

```text
Agent 不应武断决定记录天数。
应根据研究目标提出推荐天数，并标注理由、受访者负担和需要研究员确认的点。
```

### 18.3 什么样的题目最容易被后续报告使用

研究员认为，最终版中不应有无用题。

容易被报告使用的题目，必须满足：

- 每个题目都对应一个研究问题。
- 每个研究问题都能回到商业问题。
- 题目产出的信息能支持后续洞察、分群、场景描述、机会判断或报告证据。

Agent 规则：

```text
生成每一道题时，都必须标注：
- 它对应哪个研究问题。
- 它服务哪个商业问题。
- 它预计会产出什么可用于报告的信息。
```

如果一道题无法说明上述三点，应删除或合并。

### 18.4 如何判断一道题是否“可分析”

“可分析”不是让题目变成定量题，而是指题目答案能被研究员用于后续判断。

可分析题目的特征：

- 能收集到具体行为、场景、原因、比较、变化或评价标准。
- 能和其他受访者答案做横向比较。
- 能支持对场景、需求、阻碍、习惯、品牌关系或产品机会的归纳。
- 能回到研究问题和商业问题。

不可分析题目的常见问题：

- 只有泛泛态度，没有具体生活细节。
- 问得太抽象，答案难以归类。
- 问题和研究目标没有关系。
- 答案无法支持报告中的任何判断。

Agent 规则：

```text
不要只问“你怎么看”“你觉得怎么样”。
应追问具体经历、发生情境、选择原因、对比对象、变化过程和评价标准。
```

### 18.5 如何从受访者答案反推题目质量

如果大量受访者回答：

- “我不知道你什么意思。”
- “不太明白。”
- 回答明显偏离题目意图。

则说明题目本身可能有问题，需要改写。

但需要注意：

```text
受访者回答长或短，不一定直接代表问题好或不好。
话多话少可能和受访者表达习惯、投入程度、品类兴趣有关，不一定是题目质量问题。
```

Agent 规则：

```text
题目质量复盘时，不要用答案长短作为唯一标准。
更应关注受访者是否理解题意、是否回答到目标信息、是否频繁出现困惑或跑题。
```

### 18.6 品牌刺激物、产品概念、广告视频放 Diary 还是 IDI

判断取决于两个因素：

```text
1. 项目是否有 IDI 设计。
2. 刺激物复杂度和受访者理解负担。
```

适合放 diary 的情况：

- 图片、产品图、场景图。
- 内容直接，看完就能给初步反馈。
- 1-2 个短视频，每个约 1-2 分钟。
- 只需要收集直接反应、第一感受、简单评价。

不适合放 diary，更适合 IDI 的情况：

- 视频很长。
- 视频数量很多，例如 6 个。
- 内容复杂，需要理解、解释、比较。
- 需要研究员引导、追问、澄清。
- 需要深入讨论抽象 perception。

Diary 与 IDI 的基本分工：

```text
Digital Diary：收集浅层 fact 和即时 why。
IDI：追问深层原因、抽象认知和复杂 perception。
```

Agent 规则：

```text
如果刺激物复杂或数量多，应建议放入 IDI。
如果刺激物简单直接，可建议放入 diary。
如果没有 IDI 设计，应提示 diary 测试的受访者负担与理解风险。
```

### 18.7 仍待补充

关于“不同平台导入格式”和“最终导出字段”的补充判断：

- 当前阶段不需要为不同平台预设固定导入格式。
- 每类模块最终导出的字段也不需要完全固定。
- Agent 可以根据题目内容先判断并建议：
  - 题型。
  - 是否需要跳题。
  - 是否需要照片、视频、语音、截图或实物记录。
  - 对应研究目的。
  - 追问说明。
  - 受访者提示。
- 但这些都应是“建议字段”，不是强制字段。
- 最终决定权交给研究员。

Agent 规则：

```text
不要把输出格式写死成某个平台的固定字段。
先生成研究员可审阅的结构化建议；
由研究员决定哪些字段进入最终问卷或平台导入版本。
```
