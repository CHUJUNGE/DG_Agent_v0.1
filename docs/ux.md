# DG Agent UX

本文档描述 DG Agent 的产品 UX 设想。当前方向应保持轻量：系统必须先拿到一组 Agent 设计 DG 所必需的项目前置信息。Proposal 是首选输入方式，因为它通常能自动提取这些信息；但即使上传了 Proposal，缺失字段仍然必须由用户补齐。其他材料可选。Agent 先整理已收集信息和缺口，再直接输出 DG 初稿。后续修改主要通过用户与 Agent 的对话完成。

## 1. UX 定位

DG Agent 不是一个复杂的多页研究工作台，也不是单纯的 prompt 输入框。它应该是一个 proposal-first 的协作生成工具：

```text
上传 Proposal / 填写项目前置信息
-> 可选上传补充资料
-> Agent 提取并检查必填信息
-> 研究员补齐缺失必填项
-> Agent 生成 DG 初稿
-> 研究员对话修改
-> Agent 输出更新稿
```

核心原则：

- Proposal 是首选输入方式；系统会从 Proposal 中自动提取必填信息。
- 如果 Proposal 没有覆盖所有必填信息，用户仍需补齐缺失字段。
- 如果没有 Proposal，用户必须直接填写项目前置信息表。
- 其他材料是可选增强，不阻塞主流程。
- 不要求用户先审很多中间页面，但必须先补齐最低必填信息。
- Agent 应主动告诉用户“已经知道什么、还缺什么、哪些信息必须补齐后才能生成”。
- 初稿应尽快出现，然后通过对话迭代。
- 复杂的 designer / wording / evaluator / gold data 逻辑应尽量在后台完成，不暴露成繁琐 UX。

## 2. 主流程

### 2.1 提供 Agent 必需的项目前置信息

用户进入项目后，第一步是提供 Agent 必须知道的项目前置信息。推荐方式是上传 Proposal，但系统不能假设“有 Proposal 就等于信息齐全”。

Proposal 是首选输入，因为它通常可能包含：

- 项目背景。
- 商业问题。
- 研究目标。
- 目标人群。
- 方法设计。
- 样本和招募条件。
- 客户已有假设。
- 刺激物或品牌暴露要求。
- 输出期待。

Agent 需要这些信息，不是因为流程上“非要一个 Proposal 文件”，而是因为 DG 设计必须知道：

- 为什么做这个项目。
- 要解决什么商业或研究问题。
- 要问谁。
- 要用 DG 收集什么信息。
- 哪些内容能在前段问，哪些品牌 / 产品 / 刺激物必须后段才暴露。
- 是否有 IDI / 入户 / 后续访谈承接。
- 受访者需要记录几天、上传什么素材、完成什么任务。



用户不能在必填信息缺失的情况下直接生成 DG。系统应提供两条路径：

- 上传 Proposal，自动提取，然后补齐缺失项。
- 不上传 Proposal，直接填写项目前置信息表。

### 2.2 无 Proposal 时的项目前置信息表

这张表不是补充资料，而是 Agent 的必填信息结构。它既用于无 Proposal 场景，也用于 Proposal 提取后补齐缺失项。

建议字段：

#### 基础信息

- 项目名称。
- 品类 / 品牌。
- 研究类型：探索、持续追踪、产品创新、品牌 / 概念测试等。
- 输出语言。

#### 研究背景

- 为什么要做这个项目？
- 当前业务或品牌遇到的问题是什么？
- 客户已有假设或内部判断是什么？

#### 研究目标

- 这次 DG 最需要回答的 3-6 个研究问题。
- 最终希望支持什么决策：产品方向、品牌升级、传播、包装、渠道、用户分层等。

#### 目标人群

- 目标受访者是谁？
- 年龄、城市、生活状态、使用经验、购买经验、筛选条件。
- 是否有必须排除的人群。

#### 方法和任务

- DG 需要记录几天？
- 是否必须覆盖工作日 / 休息日？
- 是否包含购物任务、使用任务、拍照任务、视频任务？
- 是否有 IDI / 入户 / 后续访谈承接？

#### 品牌 / 产品 / 刺激物

- 是否有品牌、产品、概念、包装、广告等刺激物？
- 这些内容是否必须延后暴露？
- 哪些内容不能提前让受访者看到？

#### 平台和交付限制

- 是否有图片 / 视频 / 截图硬性要求？
- 是否有题量、天数、题型或必答限制？
- 是否有客户固定题或固定模块？

表单可以允许部分字段为空，但必须区分：

- **必须填写**：没有就无法合理生成。
- **建议填写**：没有也能生成，但会影响质量。
- **可后补**：不影响第一版初稿。

最低必填字段建议是：

- 品类 / 品牌。
- 研究背景或商业问题。
- 研究目标。
- 目标人群。
- 是否有品牌 / 产品 / 刺激物，以及是否需要延后暴露。
- DG 任务天数或默认可接受的任务强度。

如果用户只填写最低字段，Agent 可以用保守假设生成，并在初稿里明确列出风险。若最低字段缺失，则不能生成，只能继续要求补齐。

### 2.3 可选上传补充资料

必填信息补齐后，用户可以继续上传可选材料：

- Brief。
- 客户内部资料。
- Desk research。
- 产品 / 品牌 / 概念 / 包装 / 广告素材。
- 历史 DG。
- 访谈大纲。
- 用户补充说明。

这些材料不应该变成流程门槛。UX 上可以显示为：

```text
已提供项目前置信息
可选：继续补充材料，让 Agent 更准确
```

系统应自动排除 final DG / 标答文件，避免把标准答案当作输入材料。

### 2.4 Agent 信息收集摘要

Proposal 上传或前置信息表提交后，Agent 先输出一页或一个对话卡片，说明它从材料 / 表单中提取到了什么、哪些必填项仍缺失。

建议结构：

```markdown
## 我已经识别到的信息
- 品类 / 品牌：
- 目标人群：
- 研究目标：
- 关键研究问题：
- 方法和任务限制：
- 已知客户要求：
- 品牌 / 产品 / 刺激物暴露时机：

## 目前缺失或不确定的信息
- 生成前必须补齐：
- 建议补充：
- 可以暂时不补：

## 我会如何处理
- 可以直接生成初稿：
- 生成时会保守假设：
- 需要研究员确认的风险：
```

这个阶段的目标不是让用户审完整研究结构，而是让用户快速判断：

- Agent 有没有读懂 Proposal。
- Agent 是否已经拿到最低可用的前置信息。
- 哪些必填项还缺失。
- 哪些信息必须现在补齐，否则不能生成。
- 哪些问题可以先带着假设出稿。

### 2.5 必要信息补充

Agent 只问两类问题：

1. 必填字段缺失，必须补齐。
2. 会显著影响 DG 设计的问题，最多 3 个。

例如：

- 这个项目是否一定包含 IDI / 入户后访？
- 品牌或刺激物是否必须延后到最后？
- 日记记录需要几天？是否有工作日 / 休息日要求？
- 目标人群是否已有明确筛选条件？
- 是否有平台上传图片 / 视频的硬性要求？

如果信息不是必填，也不会显著影响结构，不要阻塞生成。Agent 应写明默认假设，例如：

```text
如果你不补充素材上传规则，我会默认素材可选；但研究目标和目标人群仍需要补齐后才能生成。
```

### 2.6 直接生成 DG 初稿

用户补齐最低必填信息后，Agent 直接生成 DG 初稿。

后台可以仍然执行两步：

```text
designer draft
-> wording pass
-> final DG wording
```

但用户默认看到的是一版可读、可改的 DG 稿，而不是 designer draft 和 wording draft 两个复杂产物。

输出建议包含：

- 简短项目理解。
- 核心研究问题。
- 模块结构总览。
- 完整 DG 题目稿。
- 需要研究员确认的问题。
- 简短 Agent 检核摘要。


### 2.7 对话修改

初稿出来后，主要 UX 是研究员和 Agent 对话修改。

用户可以直接说：

- 这个太长了，压缩到 5 个模块。
- 品牌出现太早了，往后放。
- 引导语和结束语还是太人机，按 gold data 风格重写。
- 保留研究逻辑，只改 wording。
- 这个模块交给 IDI，不要放 DG。
- 日记任务负担太重，改轻一点。
- 这些题太像 checklist，改成更自然的 diary 任务。
- 增加一个购物任务。
- 去掉重复题。

Agent 回复应优先给更新后的相关模块，而不是长篇解释。必要时再说明改动原因。

### 2.8 局部改写和版本

每次对话修改后，系统应保存版本。

用户需要能看到：

- 当前版本。
- 上一版。
- 修改了哪些模块。
- Agent 根据哪条用户指令修改。
- 是否影响研究逻辑、wording、品牌暴露、任务负担或平台格式。

但版本记录不应打断主流程，应作为侧边信息或历史记录存在。

## 3. 界面设想

### 首页 / 新项目页

主操作只有一个：

```text
上传 Proposal
```

辅助操作：

- 没有 Proposal，填写项目前置信息表。
- 粘贴 Proposal 文本。
- 上传补充资料。

### 材料区

显示：

- Proposal：首选输入方式，用于自动提取必填信息。
- 项目前置信息表：必填信息承载体；无 Proposal 时直接填写，有 Proposal 时用于补齐缺失字段。
- 补充资料：可选。
- 解析状态。
- 必填信息完整度。

### 信息收集卡片

Agent 自动输出：

- 已识别信息。
- 生成前必须补齐的信息。
- 建议补充信息。
- 默认假设。
- 是否可以直接出稿。

用户可以：

- 回答问题。
- 点击“直接生成初稿”。
- 上传更多资料。

### DG 稿件页

主区域显示当前 DG 稿。

建议布局：

- 左侧：模块目录。
- 中间：DG 正文。
- 右侧：对话区 / 修改指令 / Agent 检核摘要。

默认不要求用户切换 designer draft、wording draft、evaluator 等复杂视图。需要时可以展开高级信息。

### 对话修改区

对话区是主要修改入口。

可加快捷按钮，但不要让按钮取代自然语言：

- 更口语。
- 降低负担。
- 品牌后移。
- 重写引导语 / 结束语。
- 保留逻辑只改 wording。
- 合并重复题。
- 生成新版。

## 4. 后台能力

虽然 UX 简化，但后台仍然可以保留完整 Agent 逻辑。

### Designer Agent

负责：

- 理解 Proposal。
- 拆研究问题。
- 规划模块。
- 确定 Diary vs IDI 分工。
- 控制品牌 / 刺激物暴露顺序。
- 输出 Wording Handoff。

### Wording Agent

负责：

- 自然化模块标题。
- 改写引导语、题目、结束语。
- 降低 checklist 感。
- 降低受访者负担。
- 软化素材请求。
- 保留固定 About Me 开场。

### Evaluator

默认后台运行，给 Agent 和研究员提示风险：

- 研究问题覆盖不足。
- 题量过重。
- 品牌暴露过早。
- 任务重复。
- 引导语 / 结束语人机。
- HTML / 占位符 / 未命名模块。
- 固定模板被破坏。

Evaluator 的结果只在需要时展示，不要成为用户必须逐项处理的流程。

### Gold Data

Gold data 用于后台规则参考：

- 常见模块结构。
- 常见任务模式。
- 引导语 / 结束语写法。
- 媒体请求方式。
- 负担控制。

不要把 gold data 变成用户可见的复杂步骤，也不要直接复制历史题面。

## 5. 当前 Demo 对应关系

当前本地脚本可以对应这个 UX：

```text
上传 Proposal / case 文件
-> scripts/test_prompt_with_cases.py 构建 prompt preview
-> scripts/run_model_test.py 跑 designer-only
-> scripts/run_full_model_test.py 跑 designer + wording full pipeline
-> tests/model_tests/ 保存输出
-> 研究员 review
-> scripts/build_demo_loop_report.py 生成规则候选
```

当前 demo 已有：

- `case_data/Case_006` 这类本地 case。
- `src/prompt.py` designer prompt builder。
- `scripts/run_full_model_test.py` full pipeline。
- `skill/dg-questionnaire-designer`。
- `skill/dg-question-wording-editor`。
- `gold_data/` 离线统计和规则蒸馏。

当前 demo 还没有：

- 真正 Web 上传界面。
- Agent 必填信息完整度校验。
- 项目前置信息表 UI。
- 信息收集卡片 UI。
- 对话式局部修改 UI。
- 版本对比 UI。
- 平台导出 UI。

## 6. UX 红线

- 不要让用户先审一堆中间步骤才看到稿子。
- 不要把 designer draft、wording pass、evaluator 都做成强制页面。
- 不要让用户靠猜 prompt 才能完成常见修改。
- 不要在受访者题面暴露研究目的、品牌策略或内部逻辑。
- 不要把 gold data 原文直接展示成可复制模板。
- 不要把“上传了 Proposal”误判为“必填信息已完整”。
- 不要在最低必填信息缺失的情况下生成 DG。
- 不要因为非必填信息缺失就无限追问；先用表格补最低必需信息，再只追问真正影响方案的问题。
- 不要每个模块都写模板化感谢和主持人式转场。
