# Evaluation Plan - 题目设计 Agent 测评计划 v0.1

## 1. 测评目标

验证题目设计 Agent 是否真正学到了研究员逻辑，而不是只复制案例表面结构。

重点评估：

- 是否理解商业问题。
- 是否正确拆解研究问题。
- 是否设计合理模块。
- 是否生成有研究价值的题目。
- 是否控制受访者负担。
- 是否知道哪些地方需要研究员确认。

## 2. 当前 Case 使用策略

目前共有 4 个 case：

- Case_001 Gum：已用于规则原型和访谈沉淀。
- Case_002 Chocolate：用于补充食品饮料、零食场景、情绪驱动、品类增长规则。
- Case_003 Wonton：用于持续聆听、变化型题目、筛选核心样本，也适合作为测评 case。
- Case_004 45+ Health：用于特殊人群、健康信任、Diary + IDI 分工规则。

## 3. 第一轮建议

```text
学习/规则沉淀：Case_001, Case_002, Case_004
测评：Case_003
```

测试方式：

1. 给 agent 输入 Case_003 的 Brief / Proposal / 客户资料。
2. 不给 Final Digital Diary DG。
3. 让 agent 输出题目设计方案。
4. 与真实 Final DG 对比。

## 4. 轮换测评

后续可以轮换：

```text
Round A:
学习：001, 002, 004
测试：003

Round B:
学习：001, 003, 004
测试：002

Round C:
学习：001, 002, 003
测试：004
```

不建议当前就用全部 case 做学习，否则无法判断泛化。

## 5. 测评维度

### 5.1 项目理解

评分 1-5：

- 是否识别商业问题。
- 是否理解研究目标。
- 是否识别目标人群和品类特性。
- 是否区分客户已知信息和待探索信息。

### 5.2 研究问题拆解

评分 1-5：

- 是否把商业问题拆成消费者层面的研究问题。
- 是否遗漏关键研究问题。
- 是否加入无关研究问题。

### 5.3 模块结构

评分 1-5：

- 模块是否完整覆盖研究问题。
- 模块顺序是否合理。
- 是否知道哪些模块应保留、合并或定制。
- 是否避免机械套用 Case_001 结构。

### 5.4 题目质量

评分 1-5：

- 题目是否具体、自然、中性。
- 是否能引导真实生活细节。
- 是否避免诱导和品牌提前暴露。
- 是否有合理追问。
- 是否控制题量。

### 5.5 品类与人群适配

评分 1-5：

- 是否贴合品类特性。
- 是否贴合目标人群语气和表达负担。
- 是否使用准确的品类语言。

### 5.6 可报告化

评分 1-5：

- 每道题是否对应研究问题。
- 答案是否能支持后续报告洞察。
- 是否能产出场景、需求、阻碍、习惯、品牌/产品机会等分析材料。

### 5.7 自检与研究员确认

评分 1-5：

- 是否指出不确定点。
- 是否明确需要研究员确认的设计选择。
- 是否识别题量、刺激物、Diary vs IDI 等风险。

## 6. 对比方法

不要求 agent 输出和 Final DG 完全一致。

重点比较：

- 核心模块是否相近。
- 研究问题是否覆盖。
- 关键题目逻辑是否相近。
- 是否产生同样可分析的信息。
- 是否有更好的替代问法。

## 8. Demo 阶段通过标准

第一版 demo 可接受标准：

- 能正确识别项目类型和核心研究目标。
- 能生成合理模块结构。
- 至少 70% 模块与研究员参考思路一致或可解释。
- 题目不明显跑偏、不大面积诱导、不暴露品牌。
- 能输出需要研究员确认的问题。

## 9. 后续扩展

当 case 数量增加到 10-20 个后：

- 固定 20%-30% case 不进入规则库，只做测评。
- 建立更系统的 case card 检索。
- 对每次失败更新 research_rules。
- 逐步形成自动评分 + 人工复核的评估流程。
